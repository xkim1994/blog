JavaScript设计模式之观察者模式

发表于 2020-04-24 更新于 2021-07-15 分类于 DesignPattern

观察者模式

观察者是一种设计模式，其中一个对象（称为主题）维护一个依赖于它的对象（观察者）的列表，并自动将状态的任何更改通知它们。

当主题需要将一些有趣的事情通知观察者时，它会向观察者广播一个通知（可以包括与通知主题相关的特定数据）。

当我们不再希望某个特定的观察者通过其注册的主题而收到有关更改的通知时，该主题可以将其从观察者列表中删除。

回顾已公开的设计模式定义（通常与语言无关），以便随着时间的流逝更广泛地了解其用法和优势，通常是很有用的。GoF 书《设计模式：可重用的面向对象软件的元素》中提供的 Observer 模式的定义为：

“一个或多个观察者对主题的状态感兴趣，并通过附加自己的身份向主题注册。当观察者可能对我们的主题有所变化时，将发送一条通知消息，该通知消息在每个主题中调用 update 方法观察者。当观察者不再对主题的状态感兴趣时，他们可以干脆脱离自己。”

现在，我们可以扩展我们通过以下组件来实现观察者模式的知识：

  1. 主题：维护观察者列表，便于添加或删除观察者
  2. 观察者：为需要通知对象状态变化的对象提供更新接口
  3. ConcreteSubject：向观察者广播有关状态变化的通知，存储 ConcreteObservers 的状态
  4. ConcreteObserver：存储对 ConcreteSubject 的引用，为 Observer 实现一个更新接口，以确保状态与 Subject 的状态一致

观察者与发布/订阅模式之间的差异

尽管通常在 JavaScript 世界中，观察者模式很有用，但我们会发现它通常使用称为“发布/订阅”模式的变体来实现。尽管非常相似，但这些模式之间还是有差异的，值得注意。

观察者模式要求希望接收主题通知的观察者（或对象）必须将此兴趣订阅引发事件的对象（主题）。

但是，发布/订阅模式使用主题/事件通道，该通道位于希望接收通知的对象（订阅者）和触发事件的对象（发布者）之间。该事件系统允许代码定义特定于应用程序的事件，这些事件可以传递包含订户所需值的自定义参数。这里的想法是避免订阅者和发布者之间的依赖性。

这不同于“观察者”模式，因为它允许任何实现适当事件处理程序的订阅者注册并接 ​​ 收发布者广播的主题通知。

优点

观察者和发布/订阅模式鼓励我们认真思考应用程序不同部分之间的关 ​​ 系。它们还帮助我们确定哪些包含直接关系的图层可以替换为一组主题和观察者。这可以有效地将应用程序分解为更小，更松散耦合的块，从而改善代码管理和重用潜力。

使用 Observer 模式的进一步动机是，我们需要在不使类紧密耦合的情况下保持相关对象之间的一致性。例如，当一个对象需要能够通知其他对象而无需对这些对象进行假设时。

使用任何一种模式时，观察者和对象之间都可能存在动态关系。这提供了很大的灵活性，当我们的应用程序的各个部分紧密耦合时，可能不那么容易实现。

尽管不一定总是解决每个问题的最佳方法，但是这些模式仍然是设计解耦系统的最佳工具之一，在任何 JavaScript 开发人员的工具带中，它们都应被视为重要的工具。

缺点

因此，这些模式的某些问题实际上源于它们的主要好处。在“发布/订阅”中，通过将发布者与订阅者分离，有时可能难以获得保证应用程序的特定部分正常运行的保证。

例如，发布者可以假设一个或多个订阅者正在收听它们。假设我们使用这样的假设来记录或输出有关某些应用程序过程的错误。如果执行日志记录的订户崩溃（或由于某些原因而无法运行），由于系统的分离性质，发布者将无法看到这种情况。

这种模式的另一个缺点是订户对彼此的存在一无所知，而对更换发行人的成本却视而不见。由于订阅者和发布者之间的动态关系，更新依赖关系可能难以跟踪。

前端中那些运用的场景？

发布/订阅非常适合 JavaScript 生态系统，这在很大程度上是因为 ECMAScript 实现的核心是事件驱动的。在浏览器环境中尤其如此，因为 DOM 使用事件作为脚本的主要交互 API。

就是说，ECMAScript 和 DOM 都不提供用于在实现代码中创建自定义事件系统的核心对象或方法（可能是 DOM3 CustomEvent，该事件绑定到 DOM，因此并不通用）。
